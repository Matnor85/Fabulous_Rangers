# Gruppreflektioner

Svara på var och en av frågorna nedan som grupp. MINST 100 tecken per fråga.

## Frågor

### Inledning

#### Sammanfatta projektets syfte, samt era gemensamma mål

Syftet med denna uppgift var att kunna arbeta agilt som en grupp och på så vis bygga en hemsida med den grundkunskap vi har fått, utgående från en Figma-skiss vi hade fått tilldelad till oss.

#### Hur upplevde teamet projektet initialt?

Vi var taggade när projektet startade. Vi upplevde att det var annorlunda gentemot det vi var vana med sen innan, det med att arbeta agilt med daily stand-ups. Vi upplevde att det var lätt att dela ut arbetsuppgifter som innebar en jämn arbetsbörda över teamet utan krångliga beroenden mellan komponenter.

### Planering och genomförande

#### Beskriv hur ni inom teamet planerade projektet initialt

Vi började med att skriva Users Stories för hela Figma-skissen, och sedan bröt vi ner dem i sub-tasks. Efter det estimerade vi Story Points för dessa och prioriterade dem på så vis att kunden skulle få värdefulla inkrement i ett tidigt skede.

Där det fanns funktionell redundans lät vi den mindre värdegivande komponenten vänta till längre fram, till exempel innebar footern:s kontakt-väg att Contact Us-formen kunde vänta länge utifrån vårt perspektiv.

#### Vilka steg har teamet tagit för att utveckla applikationen?

Särskilt för vår första sprintplanering var att den största komponenten, Booking-sidans rums-presentation, bröts ned i värde-givande increment för sig.

Vi hade tre sprintar under en vecka var, och genomgående för alla sprintar var att vi genomförde Planning Poker för Story Points-estimering och prioriterade var sidas eller sid-omspännandes komponents innehåll utefter vad vi ansåg var passande för increments.

Vi startade dagarna med en daily stand-up. Under dagarna visade vi emellanåt vår progress för varandra för feedback. Fick vi en blocker tog vi upp problematiken med de övriga.

Vi reviewa:de varandras tasks kontinuerligt och litade på att vi brukade valideringsverktyg för HTML och CSS, i enlighet med en Definition of Done vi hade satt upp.

Inför sprint reviews siktade vi på att ha pusha:t ett eller fler increment var, och efter första veckan sattes målet att det skulle ske innan slutet av arbetstimmarna på torsdagen för att försäkra leverans och minska stress.

### Teamets utmaningar och lösningar

#### Vika var de största utmaningarna för teamet?

1. Samtidigt oerfaret bruk av VS Codes Github UI och dess Github-terminal verkade kanske ställa till problem med merge:ande och ledde
   dessutom till oönskat push:ande.
2. Till följd av ovannämnda oönskat pushande användes terminalen för att backa vårt repos Git history, och därefter sync:ades inte vår
   lokala history korrekt.
3. Det hände att vi rådfrågade varandra om HTML semantics och inte var säkra på vad som passade bäst här och var.

#### Hur löste eller hanterade teamet dessa utmaningar?

1. Vi höll oss till VS Code för commit:ande och push:ande.
2. Vi klonade alla ned projektet på nytt, varpå historiken var enligt repo:t.
3. Vi kollade Mozillas HTML-dokumentation och frågade ChatGPT om vad som vore passande.

#### Vilka kompromisser inom teamet har ni gjort under projektets gång?

Vi gjorde inga kompromisser. Det närmsta vi kom var att vi inte omedelbart höll med varandra om estimates från Planning Poker, men vi nådde snabbt konsensus utan missnöje (vi lade dock i de flesta fall väldigt lika estimate och ett fåtal gånger lade vi alla tre t.o.m exakt samma estimat).

Visionen var så klar i och med Figma-skissen och den handledning vi hade, så den var lätt att enas om.

### Teamets reflektion och utvärdering

#### Vad lärde sig gruppen inom projektet?

Vi lärde oss och/eller fick begrunda hur man som team kan bygga en hemsida tillsammans på så vis att man delar på CSS, att troubleshoot:a en konflikt mellan två personers CSS som team.

Vi blev mer erfarna inom Git.

#### Hur kan gruppen förbättra sitt samarbete framöver?

Vi borde uppdatera Kanban-board:en mer precist vad det gäller när saker är in-progress eller ej.

Vi tyckte oss ha en bra uppfattning huruvida vi mötte våra Story Points-målsättningar, men vi borde framöver se till att ha exakt koll på mängden Story Points vi klarat av per sprint, för att mer exakt utvärdera vår förmåga att lägga bra estimat och mer realistiskt tillsammans planera vad individer kan hinna med under en sprint.

Vi skulle kunna använda Git branches för varje komponent för att ha bättre koll på vad en persons commits hör till.
